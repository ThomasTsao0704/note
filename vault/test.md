# 🧠 程式碼提示詞設計手冊  

## Engineering Prompt Playbook

> **這不是教你寫 Prompt，而是教你如何「與工程思維對話」。**

---

### 🎯 本手冊適合誰？

- 想把 AI 當「工程夥伴」而不是聊天機器
- 常寫程式、Debug、做專案、重構系統
- 想讓 AI 產出 **可用、可維護、可擴充** 的程式碼

---

### 🧭 使用方式建議

- 🔹 Notion：章節可折疊，當成 SOP
- 🔹 Obsidian：搭配雙向連結，變成知識庫
- 🔹 GitHub：作為專案 README / Prompt 指南
- 🔹 VS Code：邊寫程式邊查提示模板

# 🧩 Chapter 0｜使用前的工程原則

> **Prompt 寫不好，不是 AI 笨，是需求不清楚。**

在任何程式相關提示前，請先想清楚四件事：

1. **語言 / 框架 / 執行環境**
2. **輸入是什麼？輸出要變成什麼？**
3. **你已經有什麼（程式碼 / 結構 / 資料）**
4. **這是 Demo、練習，還是可上線版本？**

---

# 使用前原則

AI 生成程式碼品質的 80%，取決於你提供的 20% 前置資訊：

- 指定語言＋框架（Python/FastAPI、JS/React…）
- 說明輸入什麼、輸出什麼
- 提供現有程式碼或資料結構
- 指定你要「Demo」還是「可上線」版本
